## Revision 2026-07-05 — CMS Foundation Phase C: version history and editorial workflow

**Trigger:** `19_CMS_FOUNDATION.md` §14 Phase C — build version history, the approval queue, and the version-diff UI on top of Phase B's generic engine (permissions, CRUD, `CmsForm`/`DataTable`, proven against Case Study). This revision is implementation, not redesign: every addition below is either a new, additive file (`VersionHistory` model, `lib/cms/version-history.ts`, `lib/cms/diff.ts`) or a small, sanctioned extension of an existing escape-hatch pattern (`CollectionConfig`'s `publishGuard`/`computedFields`/`deleteGuard` precedent). No existing Phase A/B file was rewritten; three were extended and one UI bug was fixed.

### What was built, matching §9 exactly

`VersionHistory` (`models/version-history.ts`) uses the literal shape `11_DATABASE_ARCHITECTURE.md` §1 specifies (`{_id, collection, documentId, snapshot, editedBy, editedAt}`) — no additional fields invented. `publish()` (`crud-actions.ts`) now snapshots the document's pre-mutation state before every publish, exactly once, in the one function that both versions and publishes (§9's "no second code path that publishes without versioning"). Restore (`restoreVersion`, new) repopulates the document's authored fields from an old snapshot and sends `status` back to `"draft"` — it does not touch `version` and does not itself write a new `VersionHistory` entry, so it goes through `submitForReview`/`publish` again like any other draft, per §9's explicit rejection of "overwrite-and-republish." The review queue (`/studio/review`) and the dashboard's "Awaiting review"/"Recent activity" cards (§10 items 2–3) are generic over `listCollections()` — Case Study is the only collection populating them today, but neither screen names Case Study anywhere in its code.

### Two small, additive `CollectionConfig` fields

Genericity required two fields the original §5 `CollectionConfig` interface didn't anticipate: `label` (a display name for section headers) and `studioBasePath` (the collection's `/studio/**` URL segment, so the review queue and activity feed can deep-link to a document's real edit page rather than only its generic history view). Both are additive to the type (one existing collection, Case Study, was updated to supply them) and follow the same shape as the existing `publishGuard`/`revalidatesPaths` escape hatches — not a new mechanism. `recordLabel` (optional; falls back to `searchableFields[0]`) was added for the same reason. A resource-generic version-history route (`/studio/history/[resource]/[id]`) and a resource-generic `restoreVersion` Server Action (`actions/studio/version-history.ts`) round out the cross-collection surface — both work for any future collection the moment it calls `registerCollection()`, with zero new code, which is the genericity check `19` asks for.

### Two Phase B gaps closed, not redesigned

1. **`useAutosave`/`autosaveDraft` existed but were never wired together** (flagged during this phase's required Phase B review) — `edit-case-study-form.tsx` now calls `useAutosave`, gated to `status === "draft"`, with a visible saved/saving/error indicator per `12_ADMIN_PANEL_SPECIFICATION.md` §4. Required a new `onValuesChange` prop on `CmsForm` (additive, optional) so a parent can observe live form state without `CmsForm` needing to know what autosave is.
2. **The single-collection "Publish" button was hidden once `status === "published"`** (`case-study-workflow-actions.tsx`), which made it impossible to ever republish an already-live document — silently defeating §9's model of `publish()` as the repeatable, only versioning path. Fixed by showing "Publish"/"Republish" (with distinct confirmation copy) for any status; the generic `publish()` action itself already had no such restriction.

### An architecture tension surfaced, not resolved — flagged for before Phase G

Restoring a **currently-published** document sends its `status` back to `"draft"`, per §9's literal text. In this system's single-document model (no separate drafts/staging collection, §6), that status flip is visible to the public the moment a public page reads `status: "published"` — today that's inert (`/work/[slug]` doesn't exist yet; only the static `/work` index and one hand-written page do, confirmed during this phase's manual QA), but it will not be inert once `14_IMPLEMENTATION_ROADMAP.md`/§13's migration (Phase G) ships a dynamic `/work/[slug]` reading live from `CaseStudy`. At that point, restoring a published Case Study will 404 its public page for the duration of the review window, which is a real, visible product behavior the founder should explicitly confirm as intended (it matches §9's literal text — restore must not silently bypass review — but "the page disappears" is a stronger effect than "the page keeps showing old content while a draft is reviewed," which is what "must not overwrite live content" reads as to a first-time reader). Recommendation: decide this explicitly before Phase G, not after the first real restore-of-a-published-page incident. Not fixed in this revision because fixing it would mean introducing a staging/shadow-document mechanism `19` §6 explicitly chose not to build.

### What was deliberately left unchanged

- **No optimistic concurrency control** (`update()`/`autosaveDraft()` remain unconditional overwrites) — already named as an accepted risk in `19` §13 with its own recommended mitigation (compare `updatedAt`); still not needed at this team's scale, and version history's snapshot-before-mutate guarantee means no edit is ever unrecoverably lost even without it.
- **No Mongoose transactions** — `snapshotVersion()` writes before the live document's mutation, deliberately, so the one failure mode a missing transaction leaves open is a harmless orphaned version entry, never a silent, unversioned overwrite.
- **The `"approve"` permission action** (granted in `roleGrants` on several resources, `permissions.ts`) remains unenforced by any distinct status/transition — `19` never specified a fourth workflow status beyond draft/review/published, and this phase's brief was explicit that Draft/Review/Published is the whole model. Left as-is; worth a future decision if a distinct approval record (not just "has publish permission") becomes a real requirement.

## Revision 2026-07-04 — Four-pillar company structure (Work / Builds / Labs / Blueprints)

**Trigger:** HubZero's operating structure evolved from client-delivery-only to four pillars — Work (unchanged), Builds (new), Labs (generalized from an existing interim mechanism), and Blueprints (new). `ARCHITECTURE/00`–`16`, all founder-approved as of 2026-07-01, accurately described the company as it existed before this evolution; none of them anticipated it. This revision is an **evolution of the existing spec, not a rewrite** — every change below is additive or narrowly targeted at the sections that actually depended on the old, narrower company definition.

### What changed, and why

**New: `17_COMPANY_STRUCTURE.md`.** The canonical definition of the four pillars — what belongs in each, what doesn't, the project lifecycle (Idea → Labs → Builds → real-world experience → Blueprints improve → Client Work → back to Labs), and governance guidance for future pillars, content types, and nav decisions. Why a new document rather than folding this into an existing one: no existing document owned "what is HubZero's operating structure" as a question — `01_PRODUCT_VISION.md` owns *positioning*, `03_INFORMATION_ARCHITECTURE.md` owns *routing*, but neither owned the underlying organizational model both of them now need to reference. **Affected:** new file only; referenced by `00`, `01`, `03`, `05`, `06`, `07`, `09`, `11`, `14`.

**`00_FOUNDER_APPROVAL.md` §8 (new).** Logs the four-pillar adoption as a founder decision, per this document's own stated convention that future decisions are recorded as amendments here, not silent edits elsewhere. Resolves three of `17`'s four open questions directly (nav ceiling, Labs/Hardware-page duplication, Blueprints framing) and leaves the fourth (Builds content readiness) explicitly open, since it's a factual question about current internal work this document can't answer on its own. **Why here and not only in `17`:** `00` is the one document every other founder-approved doc already points to for "what changed and why" — adding the decision anywhere else would have created a second place to look for the same answer.

**`03_INFORMATION_ARCHITECTURE.md` — sitemap, nav, footer, URL/depth rules (major).** Added `/builds`, `/labs`, `/blueprints` to the sitemap; raised the primary-nav ceiling from five items to six items plus a CTA (Services, Work, Builds, Labs, Blueprints, About + Contact), with content-gated rollout (a pillar enters nav only once it has real published content) and Blog moved to footer-only to hold the line at six; added the three pillars to the footer; confirmed the existing URL and depth/breadth conventions apply unchanged. **Why:** this is the single document whose core numeric commitment ("five items plus one CTA," cited as research-backed) was directly falsified by the company having a fourth genuinely distinct visitor intent. The fix preserves the *principle* (a short nav reflecting real intent) while updating the *number* it produces — explicitly reasoned out rather than silently drifted into, per the instruction not to blindly preserve old decisions.

**`04_USER_JOURNEYS.md` — journey extensions (minor).** Added Builds and Labs as required stops in the Partner/Collaborator journey (§5) — capability breadth beyond client work is exactly what that audience needs to assess — and Labs as an optional stop in the Potential Recruit journey (§4). **Why minor, not major:** none of the five existing journeys needed replacing; two needed one more required page each.

**`05_CONTENT_STRATEGY.md` — new content types and quality bars (major).** Added Builds and Blueprints to the content-types table; generalized the Labs/R&D row from "interim hardware-capability proof" to "permanent pillar covering hardware, software, and AI"; added new §2a (Builds content policy) and §2b (Blueprints content policy), the latter introducing a genuinely new kind of honesty risk this document hadn't needed before — a Blueprint's live demo is a checkable claim a visitor can falsify in one click, unlike an editorial claim that's merely unverifiable. **Why:** this document is where the zero-fabrication policy lives, and extending it explicitly to two new content types (rather than assuming it applies by default) matches how every other addition to this document has been handled — stated, not implied.

**`06_PAGE_SPECIFICATIONS.md` — three new page specs, one rewritten (major).** Added full specs for Builds, Labs, and Blueprints (index + detail each), each with an explicit "compositional difference" note tying back to the design-review finding below. Rewrote the Hardware & Embedded page's Labs/R&D section from a full embedded write-up into a summary-plus-link to the new canonical `/labs/[slug]` entry. **Why the rewrite, not just an addition:** leaving the old embedded write-up in place alongside the new top-level pillar would have created exactly the kind of two-sources-of-truth problem this codebase has already been burned by once (the legacy `team.json` duplication) — this was fixed pre-emptively rather than left as a known risk.

**`07_DESIGN_SYSTEM.md` §8 (new).** Generalizes `15_HOMEPAGE_DESIGN.md` §12's per-beat Uniqueness Test from a homepage-only tool into a standing requirement for any new marketing page, and assigns editorial-vs-utility component vocabulary to the three new pillar pages (editorial for detail pages, a lighter utility treatment acceptable for index/filter controls, matching the existing Work index pattern). **Why now, specifically:** the commissioned design review (`docs/design-reviews/MARKETING_SITE_REVIEW_V1.md`, 2026-07-04) found Services/Software/Hardware collapsed into "three fills of one template" precisely because the homepage's own Uniqueness Test was never applied past the homepage. Three more index/detail page families were about to be built under the same unstated assumption — this closes that gap before it repeats at twice the scale, rather than after.

**`09_CMS_ARCHITECTURE.md` §2 (major).** Added Builds and Blueprints collection rows; updated the Labs/R&D row's description to match its new permanent-pillar status. RBAC and workflow model (§3-4) required no change — both were already generic across collections, which is exactly why adding two more collections cost a documentation update and not a re-architecture.

**`10_FEATURE_SPECIFICATION.md` §3 (minor).** Added Builds and Blueprints to the "new, did not exist in legacy" list, alongside the existing Labs/R&D entry (updated to note its generalization).

**`11_DATABASE_ARCHITECTURE.md` §1, §2, §5 (major).** Added `Build` and `Blueprint` collection schemas; added `stage` and `graduatedToBuildId` fields to `LabsProject`, and the inverse `graduatedFromLabsId` field to `Build`, so the lifecycle in `17` §3 is a real, bidirectionally-validated relationship rather than a narrative claim; extended the shared `practiceArea` enum to include `'ai'` (already permitted by `00_FOUNDER_APPROVAL.md` §6's existing extensibility clause, exercised here for the first time); added indexing for the two new collections. **Why the graduation fields specifically:** without them, the first real Labs→Builds graduation would either lose its own history or require a manual, error-prone reconstruction later — cheaper to model the relationship before it's needed than to retrofit it after the first graduation already happened.

**`12_ADMIN_PANEL_SPECIFICATION.md` §2 (minor).** Added Builds and Blueprints screens; updated the Labs/R&D screen description to reflect its top-level pillar status and added a "mark as graduated" action.

**`13_SEO_STRATEGY.md` §2, §3 (minor).** Added Build/Labs/Blueprint rows to the per-page-type metadata table — Blueprint specifically needed a different structured-data treatment (a Product/CreativeWork hybrid) than the Article/CreativeWork pattern used for case studies and blog posts, since it's the one content type with a category and a live demo. Confirmed sitemap generation extends to the new collections without a separate mechanism.

**`14_IMPLEMENTATION_ROADMAP.md` §0, new Phase 3a, Definition of Done (major).** Added an explicit scope-weighing sentence in §0 (the same discipline already applied to the blog-MVP decision): Labs ships alongside Phase 3 because it's substantially a content migration; Builds and Blueprints are content-gated and treated as fast-follow candidates unless real content already exists. Inserted Phase 3a, sequencing Labs (migrate existing content) → Builds (content-gated) → Blueprints (highest build cost, new demo infrastructure) → nav promotion. Updated the Definition of Done to make Builds/Blueprints' MVP status conditional rather than assumed either way.

### What was deliberately left unchanged

- **`01_PRODUCT_VISION.md`, `02_BRAND_STRATEGY.md`** — received only a one-line cross-reference each. Positioning, ideal-client profiles, and voice rules don't depend on how many operating pillars the company has; they needed pointers to `17`, not rewrites.
- **`08_TECHNICAL_ARCHITECTURE.md`** — only the illustrative layout diagram in §5 changed. Stack decisions, rendering strategy, and the hybrid storage model are pillar-agnostic.
- **`15_HOMEPAGE_DESIGN.md`** — untouched. It's homepage-specific and already Round-2 founder-approved; the pillar restructuring doesn't change the homepage's own beats. Its Uniqueness Test is *referenced and generalized* from `07` §8, not edited in place — `15` remains the original, narrowly-scoped source of that idea.
- **`16_RESPONSIVE_DESIGN_STANDARDS.md`** — untouched. It's already written to apply to "any future marketing-page section," which already covers the three new pillar pages without modification.

### A documentation debt named, not fixed, in this revision

`15_HOMEPAGE_DESIGN.md` §7/§11 still describe the Bhatkal Time Luxe case study as "MERN + Imgix, built Apr–May 2025" — already flagged as stale in `PROJECT_CONTEXT.md` §7.1, superseded by the real research doc and the real implemented page. This predates and is unrelated to the four-pillar revision; it's named here only because this session touched every other document and it would be misleading not to mention a known inconsistency in the one document this pass didn't edit. Left for a separate, dedicated fix.

### Open questions carried forward (not resolved by documentation alone)

Per `00_FOUNDER_APPROVAL.md` §8: whether a real, complete internal product exists today to publish as the first Build is a factual question, not a documentation question — `14_IMPLEMENTATION_ROADMAP.md` Phase 3a is written to be correct either way (content-gated), but the answer determines whether Builds ships alongside Labs or is deferred as a fast-follow.
