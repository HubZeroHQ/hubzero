## Revision 2026-07-06 — Editorial content blocks replace fixed markdown fields

**Trigger:** the CMS's five narrative collections (Case Studies, Labs Projects, Builds, Blueprints, Notes) each forced authors into a fixed set of named markdown fields — most visibly Case Study's mandatory `problem`/`approach`/`result` three-beat structure. A hardware bring-up log and a client web platform aren't the same kind of story; this revision replaces the fixed fields with a single ordered `content: Block[]` field per collection, so the author arranges whatever mix of building blocks (heading, paragraph, image, gallery, quote, callout, code, divider, metrics, timeline, video, spacer, two-column, markdown, raw HTML) the actual narrative needs, in whatever order. New canonical document: `20_CONTENT_BLOCKS.md` (full spec, cross-referenced from `19_CMS_FOUNDATION.md` §6's field-type vocabulary).

### Genericity held — one new field type, not a parallel system

`"blocks"` is the one addition to `FieldConfig.type` (`types/cms.ts`), following the exact precedent `"json"` set in the Phase D collections rollout: a closed-vocabulary field type, not a plugin registry. `crud-actions.ts`'s `rawFromFormData`/`create`/`update`/`autosaveDraft` needed zero changes beyond one generic addition (the Raw HTML publish guard, below) — a `content: Block[]` field is, to the engine, just another opaque field value carried as a JSON string over the wire (`lib/cms/blocks/schema.ts`'s `blocksField()`), the same wire contract `"json"` already established. `getMediaUsage()` (`lib/cms/media.ts`) gained one generic extension: scanning any `"blocks"`-type field's content for embedded `Media` ids (`lib/cms/blocks/guard.ts`'s `collectBlockMediaIds`), so "which documents use this image" stays correct for an image referenced only inside a block.

### What was added, beyond the block field itself

- **Card metadata** (`ARCHITECTURE/20_CONTENT_BLOCKS.md` §3): `summary` (Case Study, Labs Project, Blueprint — Build's `tagline`/Note's `summary` already served this role), `featured` (all five), `readingTimeMinutes` (generalized from Note-only to all five, computed from `content`'s word count).
- **Team relationships** (§4): `contributors: TeamMember[]` on all five collections, via the existing generic `referenceArray` field type and searchable picker — zero new picker code needed.
- **Homepage feature system** (§6): `SiteSettings.featuredCaseStudyId` (optional) plus a fallback chain (explicit pick → most recent `featured: true` → most recent published) replaces the hardcoded "Bhatkal Time Luxe" homepage component (`components/marketing/case-study.tsx`, now an async Server Component reading real CMS data).
- **Generic public renderer** (§7): `<ContentRenderer>`/`<BlockRenderer>` (`components/marketing/blocks/`) replace the old fixed `<RichText>{doc.problem}</RichText>`-style sections on `/work/[slug]`, `/labs/[slug]`, `/blueprints/[slug]`, `/notes/[slug]` — each page shrank to one renderer call plus contributor chips.
- **Generic admin block editor** (`components/admin/blocks/`): reorder (via `motion`'s `Reorder.Group`, already a dependency — no new drag-and-drop library added), collapse/expand, duplicate, delete, and an "Add block" picker grouped by purpose. One implementation, used identically by all five collections' `content` field.

### The Raw HTML block's "admin-only" rule is a publish-time guard, not a save-time one

Rejecting an `html` block at save time would brick a Teammate's in-progress draft the moment they experiment with the block. Instead, `crud-actions.ts`'s `publish()` gained a generic `checkBlocksPublishGuard()` — introspects `formFields` for any `"blocks"` field and blocks publish (with a clear message) unless the publishing user is Admin/Head Admin — composing alongside a collection's own `publishGuard` (Blueprint's `demoStatus` gate), not a second guard mechanism.

### Migration — idempotent, nothing lost

`scripts/migrate-content-blocks.ts` converts every pre-existing document's legacy fields into `content`, via pure, unit-tested transforms (`lib/cms/blocks/legacy-migration.ts`): Case Study's `problem`/`approach`/`result` each become a heading+markdown pair; Build/Labs Project/Note's single field becomes one markdown block; Blueprint's `description`+`customizationNotes` become a markdown block plus an optional heading+markdown pair. `needsContentMigration()` makes re-running the script a safe no-op for already-migrated documents; old fields are `$unset` only after `content` is written successfully.

### What was deliberately not built, and two open founder decisions

No syntax highlighting for the Code block (would need a new dependency — plain monospace matches the existing `RichText` inline-code treatment), no structured per-block version-history diff (the existing generic JSON-diff fallback already renders it, just not beautifully), no rotating/multiple featured Case Studies (the schema is shaped for an easy additive upgrade later). Two fields named as *examples* in the brief — a card `accentImage` distinct from `coverImage`, and a Notes `difficulty` field — were deliberately not invented without a concrete design for what either means; flagged for founder decision in `ARCHITECTURE/20_CONTENT_BLOCKS.md` §11 rather than built speculatively.

## Revision 2026-07-05 — "Blog" renamed to "Notes"

**Trigger:** HubZero publishes engineering write-ups, architectural thinking, dev logs, experiments, lessons learned, and product thinking — not a traditional marketing blog. "Blog" never matched what this content type actually is or was ever going to be, and no public route had shipped under that name, so this is a clean rename with no backward-compatible URL or redirect concerns. This revision renames the content type/collection/section from "Blog" to "Notes" throughout the architecture set — model, CMS collection, admin routes (`/studio/notes`), and public routes (`/notes`, `/notes/[slug]`) — without changing its scope, workflow, or editorial bar in any way. Every prior document reference to "Blog"/"Blog Posts"/"blog platform" is updated in place to "Notes"; only entries in this changelog and `00_FOUNDER_APPROVAL.md` that narrate a specific past session's events (e.g. the Phase D entry immediately below, dated the same day) keep their original "Blog" wording, since they are a record of what was true when they were written, not living architecture. **Affected:** `00`, `01`, `03`, `05`, `06`, `07`, `08`, `09`, `10`, `11`, `12`, `13`, `14`, `17`, `19`, `PROJECT_CONTEXT.md`, `README.md` — documentation only; the corresponding code rename (models, CMS collection config, Server Actions, admin routes, permissions, nav config) is a separate, concurrent change.

## Revision 2026-07-05 — CMS Foundation Phase D: the remaining eight collections

**Trigger:** `19_CMS_FOUNDATION.md` §14 Phase E ("roll out the remaining ten collections... each is now `defineCollection({...})` plus a Mongoose model plus (where needed) a small collection-specific extension") — the payoff phase for Phase B's engine investment. This revision implements Team, Testimonials, FAQ, Career Listings, Labs Projects, Builds, Blueprints, and Blog Posts (Services and Settings remain out of scope; see below) entirely on the Phase B/C generic engine. Two small, sanctioned engine additions were required (a new field type, a configurable ownership field); three UI components were generalized from Case-Study-only to collection-agnostic; two real bugs in the Phase B/C engine were found during manual QA and fixed. No collection needed a bespoke CRUD/permission/versioning path beyond the two explicitly anticipated in `19` §11 (Labs' graduation action, Blueprint's publish guard).

### Genericity held — zero collection-specific engine forks

Every one of the eight collections is a `defineCollection({...})` config plus a Mongoose model, consumed by the unmodified `createCrudActions()`, `DataTable`, `CmsForm`, review queue, version-history screen, and dashboard activity feed. The `"Did this require modifying the CMS engine?"` check (this phase's own mandate) was applied after every collection; the answers below are the only two "yes"es out of eight, and both were judged to benefit future collections, not just the one that surfaced them:

1. **A `"json"` field type** (`types/cms.ts`, `CmsField`) — the escape hatch for `TeamMember.skills`/`experience`/`education`, the first fields needing a nested array-of-objects shape the fixed vocabulary (text/richtext/select/multiselect/image/reference/etc.) can't express. A raw-JSON textarea, validated by the collection's own Zod schema (`lib/cms/json-field.ts`'s `jsonArray()` helper) — deliberately not a structured repeater UI, matching the same "plain input until a real picker lands" precedent `reference` already set in Phase B.
2. **`CollectionConfig.ownerField`** (`collection-config.ts`, read by `crud-actions.ts`'s `ownerTarget()`) — `editOwn` needs to compare against *whose content this is*, not *who created the database row*. For every other collection those are the same person (the creator is the author); for `TeamMember` they aren't (a Head Admin typically creates a Teammate's profile row during onboarding). Without this, the teammate a profile belongs to could never edit their own bio, and whichever admin onboarded them could edit it forever — a real functional bug blocking `09_CMS_ARCHITECTURE.md` §3's "Teammates can create and edit their own... portfolio content freely," not a theoretical one. `TeamMember.linkedUserId` is the field `19_CMS_FOUNDATION.md` §11 already named as the required addition; this phase just wires it into the ownership check.

Two smaller, purely-additive precedents: `practiceAreaOptions` (shared by CaseStudy/Build/LabsProject per `11_DATABASE_ARCHITECTURE.md` §1's own "this enum is shared" note) moved from `case-study-fields.tsx` into `lib/cms/collections/shared-options.ts` before a second and third copy could exist; `objectIdField`/`optionalObjectIdField` (`shared-validation.ts`) centralize the one validation shape every `reference` field needs.

### Three components generalized from Case-Study-only to every collection

Building the second through ninth collection on top of the first surfaced three files that were Case-Study-specific only because Case Study had been the only collection: `case-study-workflow-actions.tsx` (submit/publish/delete controls), `new-case-study-form.tsx` (create-and-redirect), and `edit-case-study-form.tsx` (autosave-wired edit). Each is now a shared component (`components/admin/workflow-actions.tsx`, `cms-create-form.tsx`, `cms-edit-form.tsx`) parameterized by props instead of importing one collection's actions directly; Case Study was retrofitted onto all three in the same pass, so nothing collection-specific was left behind as an inconsistent snowflake. `lib/cms/table-search-params.ts` extracts the identical `searchParams`-parsing boilerplate every list page's Server Component needs, previously duplicated verbatim in `case-studies/page.tsx`.

### Two real Phase B/C engine bugs found during this phase's manual QA, fixed

1. **`WorkflowActions` silently discarded every action's error result.** `handlePublish`/`handleSubmitForReview`/`handleDelete` all checked `result.status === "success"` and did nothing on `"error"` — the confirm dialog closed, the document silently didn't change, and there was no way for an editor to know why. This existed in the Case-Study-only predecessor too; it went unnoticed because Case Study's `publish()` has no `publishGuard` to ever fail in normal use. Blueprint's `demoStatus !== "live"` guard is what actually exercised the failure path for the first time. Fixed with local `actionError` state rendered as an `<Alert>` — one fix, every collection benefits.
2. **The generic diff view rendered a nested object as `"[object Object]"`.** `diff-view.tsx`'s `formatValue()` handled primitive arrays (`.join(", ")`) and scalars, but not a plain object or an array of objects — `TeamMember.socials` (and, once populated, `skills`/`experience`/`education`) is the first field shape to hit this. Fixed generically (`JSON.stringify` for the object/object-array cases), not special-cased to `socials`.

### An architecture tension found and resolved by disambiguation, not by picking a side

`ARCHITECTURE/11_DATABASE_ARCHITECTURE.md` §1's `CareerListing` block reads `status: 'open' | 'closed'` — a domain concept (is this role still accepting applicants), the same shape as `Lead.status`. But `09_CMS_ARCHITECTURE.md` §2's workflow column says "Draft → Published" for Career Listings, and `19_CMS_FOUNDATION.md` §11 explicitly maps it to `workflow: "draft-publish"` alongside Testimonials/FAQs/Services — two founder-approved documents in tension over what one field name means. Resolved by keeping both concepts, disambiguated by name: `status` is the standard `workflowFields()` draft/published lifecycle (so a listing goes through the same review/version-history machinery as everything else before it's publicly visible); `listingStatus` is `11`'s open/closed concept, applying only once a listing is actually published. Every field `11` names is preserved; nothing was invented — only the collision between two approved documents' use of the same JSON key is resolved. Flagged here for founder confirmation rather than silently picked.

### Two documented assumptions, flagged rather than silently invented

- **`TeamMember.experience`/`education` item shapes.** `11_DATABASE_ARCHITECTURE.md` §1 names these fields (`experience?: ExperienceItem[], education?: EducationItem[]`) but never defines the two interfaces. `models/team-member.ts` uses a minimal, resume-style shape (free-text `startDate`/`endDate` rather than a strict `Date`, since that's how this information is actually written — "2022," "Jan 2022–Present"). Worth founder review before any public `/team` profile page renders these fields.
- **`Blueprint.blueprintId` generation.** `11` specifies the field's shape and uniqueness ("exposed in metadata, never in the URL") but not how it's generated. This phase treats it as editor-provided, like `slug` — no auto-generation scheme invented.

### What was deliberately out of scope for this phase

- **Services and Settings** — named in `09_CMS_ARCHITECTURE.md` §2 and `19` §11 but not in this phase's brief; `Settings` in particular is explicitly the one collection `19` §11 calls out as not fitting the list-of-documents shape (`findOneAndUpdate` + upsert, no `DataTable`), deferred rather than forced through machinery built for a different shape.
- **Homepage content and Privacy/Terms as CMS collections** — checked explicitly against this phase's brief ("only if ARCHITECTURE specifies CMS ownership"). Neither has a schema in `11_DATABASE_ARCHITECTURE.md`; `03_INFORMATION_ARCHITECTURE.md` and `06_PAGE_SPECIFICATIONS.md` describe Privacy/Terms as rewritten static legal copy, not CMS-authored content (contrast with Careers, whose page spec explicitly says "open roles (CMS-managed, can be empty)"). Building a collection for either would have violated this phase's "do not invent schemas" instruction. `SiteSettings` (§1's actual sitewide-content vehicle) is Services/Settings' phase, not this one.
- **Public-facing pages for any of the eight collections** — `config/nav.ts` already documents that Team/Careers/Blog/Blueprints/Labs/Builds have no shipped public route yet ("added here the same day their page ships, not before"); this phase built the Studio (admin) side only, per its own brief. `revalidatesPaths` is correspondingly `() => []` for all eight — there is nothing yet to revalidate.
- **Media library / `MediaPicker`** — `image`/`imageArray` fields render as plain URL inputs for all eight collections, identical to Case Study's existing placeholder, per `19` §14 Phase D's (the architecture document's own internal "Phase D," media library — a different numbering than this session's "Phase D," which is collections) own note that this is additive later, not a schema change now.

## Revision 2026-07-05 — CMS Foundation Phase C: version history and editorial workflow

**Trigger:** `19_CMS_FOUNDATION.md` §14 Phase C — build version history, the approval queue, and the version-diff UI on top of Phase B's generic engine (permissions, CRUD, `CmsForm`/`DataTable`, proven against Case Study). This revision is implementation, not redesign: every addition below is either a new, additive file (`VersionHistory` model, `lib/cms/version-history.ts`, `lib/cms/diff.ts`) or a small, sanctioned extension of an existing escape-hatch pattern (`CollectionConfig`'s `publishGuard`/`computedFields`/`deleteGuard` precedent). No existing Phase A/B file was rewritten; three were extended and one UI bug was fixed.

### What was built, matching §9 exactly

`VersionHistory` (`models/version-history.ts`) uses the literal shape `11_DATABASE_ARCHITECTURE.md` §1 specifies (`{_id, collection, documentId, snapshot, editedBy, editedAt}`) — no additional fields invented. `publish()` (`crud-actions.ts`) now snapshots the document's pre-mutation state before every publish, exactly once, in the one function that both versions and publishes (§9's "no second code path that publishes without versioning"). Restore (`restoreVersion`, new) repopulates the document's authored fields from an old snapshot and sends `status` back to `"draft"` — it does not touch `version` and does not itself write a new `VersionHistory` entry, so it goes through `submitForReview`/`publish` again like any other draft, per §9's explicit rejection of "overwrite-and-republish." The review queue (`/studio/review`) and the dashboard's "Awaiting review"/"Recent activity" cards (§10 items 2–3) are generic over `listCollections()` — Case Study is the only collection populating them today, but neither screen names Case Study anywhere in its code.

### Two small, additive `CollectionConfig` fields

Genericity required two fields the original §5 `CollectionConfig` interface didn't anticipate: `label` (a display name for section headers) and `studioBasePath` (the collection's `/studio/**` URL segment, so the review queue and activity feed can deep-link to a document's real edit page rather than only its generic history view). Both are additive to the type (one existing collection, Case Study, was updated to supply them) and follow the same shape as the existing `publishGuard`/`revalidatesPaths` escape hatches — not a new mechanism. `recordLabel` (optional; falls back to `searchableFields[0]`) was added for the same reason. A resource-generic version-history route (`/studio/history/[resource]/[id]`) and a resource-generic `restoreVersion` Server Action (`actions/studio/version-history.ts`) round out the cross-collection surface — both work for any future collection the moment it calls `registerCollection()`, with zero new code, which is the genericity check `19` asks for.

### Two Phase B gaps closed, not redesigned

1. **`useAutosave`/`autosaveDraft` existed but were never wired together** (flagged during this phase's required Phase B review) — `edit-case-study-form.tsx` now calls `useAutosave`, gated to `status === "draft"`, with a visible saved/saving/error indicator per `12_ADMIN_PANEL_SPECIFICATION.md` §4. Required a new `onValuesChange` prop on `CmsForm` (additive, optional) so a parent can observe live form state without `CmsForm` needing to know what autosave is.
2. **The single-collection "Publish" button was hidden once `status === "published"`** (`case-study-workflow-actions.tsx`), which made it impossible to ever republish an already-live document — silently defeating §9's model of `publish()` as the repeatable, only versioning path. Fixed by showing "Publish"/"Republish" (with distinct confirmation copy) for any status; the generic `publish()` action itself already had no such restriction.

### An architecture tension surfaced, not resolved — flagged for before Phase G

Restoring a **currently-published** document sends its `status` back to `"draft"`, per §9's literal text. In this system's single-document model (no separate drafts/staging collection, §6), that status flip is visible to the public the moment a public page reads `status: "published"` — today that's inert (`/work/[slug]` doesn't exist yet; only the static `/work` index and one hand-written page do, confirmed during this phase's manual QA), but it will not be inert once `14_IMPLEMENTATION_ROADMAP.md`/§13's migration (Phase G) ships a dynamic `/work/[slug]` reading live from `CaseStudy`. At that point, restoring a published Case Study will 404 its public page for the duration of the review window, which is a real, visible product behavior the founder should explicitly confirm as intended (it matches §9's literal text — restore must not silently bypass review — but "the page disappears" is a stronger effect than "the page keeps showing old content while a draft is reviewed," which is what "must not overwrite live content" reads as to a first-time reader). Recommendation: decide this explicitly before Phase G, not after the first real restore-of-a-published-page incident. Not fixed in this revision because fixing it would mean introducing a staging/shadow-document mechanism `19` §6 explicitly chose not to build.

### What was deliberately left unchanged

- **No optimistic concurrency control** (`update()`/`autosaveDraft()` remain unconditional overwrites) — already named as an accepted risk in `19` §13 with its own recommended mitigation (compare `updatedAt`); still not needed at this team's scale, and version history's snapshot-before-mutate guarantee means no edit is ever unrecoverably lost even without it.
- **No Mongoose transactions** — `snapshotVersion()` writes before the live document's mutation, deliberately, so the one failure mode a missing transaction leaves open is a harmless orphaned version entry, never a silent, unversioned overwrite.
- **The `"approve"` permission action** (granted in `roleGrants` on several resources, `permissions.ts`) remains unenforced by any distinct status/transition — `19` never specified a fourth workflow status beyond draft/review/published, and this phase's brief was explicit that Draft/Review/Published is the whole model. Left as-is; worth a future decision if a distinct approval record (not just "has publish permission") becomes a real requirement.

## Revision 2026-07-04 — Four-pillar company structure (Work / Builds / Labs / Blueprints)

**Trigger:** HubZero's operating structure evolved from client-delivery-only to four pillars — Work (unchanged), Builds (new), Labs (generalized from an existing interim mechanism), and Blueprints (new). `ARCHITECTURE/00`–`16`, all founder-approved as of 2026-07-01, accurately described the company as it existed before this evolution; none of them anticipated it. This revision is an **evolution of the existing spec, not a rewrite** — every change below is additive or narrowly targeted at the sections that actually depended on the old, narrower company definition.

### What changed, and why

**New: `17_COMPANY_STRUCTURE.md`.** The canonical definition of the four pillars — what belongs in each, what doesn't, the project lifecycle (Idea → Labs → Builds → real-world experience → Blueprints improve → Client Work → back to Labs), and governance guidance for future pillars, content types, and nav decisions. Why a new document rather than folding this into an existing one: no existing document owned "what is HubZero's operating structure" as a question — `01_PRODUCT_VISION.md` owns *positioning*, `03_INFORMATION_ARCHITECTURE.md` owns *routing*, but neither owned the underlying organizational model both of them now need to reference. **Affected:** new file only; referenced by `00`, `01`, `03`, `05`, `06`, `07`, `09`, `11`, `14`.

**`00_FOUNDER_APPROVAL.md` §8 (new).** Logs the four-pillar adoption as a founder decision, per this document's own stated convention that future decisions are recorded as amendments here, not silent edits elsewhere. Resolves three of `17`'s four open questions directly (nav ceiling, Labs/Hardware-page duplication, Blueprints framing) and leaves the fourth (Builds content readiness) explicitly open, since it's a factual question about current internal work this document can't answer on its own. **Why here and not only in `17`:** `00` is the one document every other founder-approved doc already points to for "what changed and why" — adding the decision anywhere else would have created a second place to look for the same answer.

**`03_INFORMATION_ARCHITECTURE.md` — sitemap, nav, footer, URL/depth rules (major).** Added `/builds`, `/labs`, `/blueprints` to the sitemap; raised the primary-nav ceiling from five items to six items plus a CTA (Services, Work, Builds, Labs, Blueprints, About + Contact), with content-gated rollout (a pillar enters nav only once it has real published content) and Blog moved to footer-only to hold the line at six; added the three pillars to the footer; confirmed the existing URL and depth/breadth conventions apply unchanged. **Why:** this is the single document whose core numeric commitment ("five items plus one CTA," cited as research-backed) was directly falsified by the company having a fourth genuinely distinct visitor intent. The fix preserves the *principle* (a short nav reflecting real intent) while updating the *number* it produces — explicitly reasoned out rather than silently drifted into, per the instruction not to blindly preserve old decisions.

**`04_USER_JOURNEYS.md` — journey extensions (minor).** Added Builds and Labs as required stops in the Partner/Collaborator journey (§5) — capability breadth beyond client work is exactly what that audience needs to assess — and Labs as an optional stop in the Potential Recruit journey (§4). **Why minor, not major:** none of the five existing journeys needed replacing; two needed one more required page each.

**`05_CONTENT_STRATEGY.md` — new content types and quality bars (major).** Added Builds and Blueprints to the content-types table; generalized the Labs/R&D row from "interim hardware-capability proof" to "permanent pillar covering hardware, software, and AI"; added new §2a (Builds content policy) and §2b (Blueprints content policy), the latter introducing a genuinely new kind of honesty risk this document hadn't needed before — a Blueprint's live demo is a checkable claim a visitor can falsify in one click, unlike an editorial claim that's merely unverifiable. **Why:** this document is where the zero-fabrication policy lives, and extending it explicitly to two new content types (rather than assuming it applies by default) matches how every other addition to this document has been handled — stated, not implied.

**`06_PAGE_SPECIFICATIONS.md` — three new page specs, one rewritten (major).** Added full specs for Builds, Labs, and Blueprints (index + detail each), each with an explicit "compositional difference" note tying back to the design-review finding below. Rewrote the Hardware & Embedded page's Labs/R&D section from a full embedded write-up into a summary-plus-link to the new canonical `/labs/[slug]` entry. **Why the rewrite, not just an addition:** leaving the old embedded write-up in place alongside the new top-level pillar would have created exactly the kind of two-sources-of-truth problem this codebase has already been burned by once (the legacy `team.json` duplication) — this was fixed pre-emptively rather than left as a known risk.

**`07_DESIGN_SYSTEM.md` §8 (new).** Generalizes `15_HOMEPAGE_DESIGN.md` §12's per-beat Uniqueness Test from a homepage-only tool into a standing requirement for any new marketing page, and assigns editorial-vs-utility component vocabulary to the three new pillar pages (editorial for detail pages, a lighter utility treatment acceptable for index/filter controls, matching the existing Work index pattern). **Why now, specifically:** the commissioned design review (`docs/design-reviews/MARKETING_SITE_REVIEW_V1.md`, 2026-07-04) found Services/Software/Hardware collapsed into "three fills of one template" precisely because the homepage's own Uniqueness Test was never applied past the homepage. Three more index/detail page families were about to be built under the same unstated assumption — this closes that gap before it repeats at twice the scale, rather than after.

**`09_CMS_ARCHITECTURE.md` §2 (major).** Added Builds and Blueprints collection rows; updated the Labs/R&D row's description to match its new permanent-pillar status. RBAC and workflow model (§3-4) required no change — both were already generic across collections, which is exactly why adding two more collections cost a documentation update and not a re-architecture.

**`10_FEATURE_SPECIFICATION.md` §3 (minor).** Added Builds and Blueprints to the "new, did not exist in legacy" list, alongside the existing Labs/R&D entry (updated to note its generalization).

**`11_DATABASE_ARCHITECTURE.md` §1, §2, §5 (major).** Added `Build` and `Blueprint` collection schemas; added `stage` and `graduatedToBuildId` fields to `LabsProject`, and the inverse `graduatedFromLabsId` field to `Build`, so the lifecycle in `17` §3 is a real, bidirectionally-validated relationship rather than a narrative claim; extended the shared `practiceArea` enum to include `'ai'` (already permitted by `00_FOUNDER_APPROVAL.md` §6's existing extensibility clause, exercised here for the first time); added indexing for the two new collections. **Why the graduation fields specifically:** without them, the first real Labs→Builds graduation would either lose its own history or require a manual, error-prone reconstruction later — cheaper to model the relationship before it's needed than to retrofit it after the first graduation already happened.

**`12_ADMIN_PANEL_SPECIFICATION.md` §2 (minor).** Added Builds and Blueprints screens; updated the Labs/R&D screen description to reflect its top-level pillar status and added a "mark as graduated" action.

**`13_SEO_STRATEGY.md` §2, §3 (minor).** Added Build/Labs/Blueprint rows to the per-page-type metadata table — Blueprint specifically needed a different structured-data treatment (a Product/CreativeWork hybrid) than the Article/CreativeWork pattern used for case studies and blog posts, since it's the one content type with a category and a live demo. Confirmed sitemap generation extends to the new collections without a separate mechanism.

**`14_IMPLEMENTATION_ROADMAP.md` §0, new Phase 3a, Definition of Done (major).** Added an explicit scope-weighing sentence in §0 (the same discipline already applied to the blog-MVP decision): Labs ships alongside Phase 3 because it's substantially a content migration; Builds and Blueprints are content-gated and treated as fast-follow candidates unless real content already exists. Inserted Phase 3a, sequencing Labs (migrate existing content) → Builds (content-gated) → Blueprints (highest build cost, new demo infrastructure) → nav promotion. Updated the Definition of Done to make Builds/Blueprints' MVP status conditional rather than assumed either way.

### What was deliberately left unchanged

- **`01_PRODUCT_VISION.md`, `02_BRAND_STRATEGY.md`** — received only a one-line cross-reference each. Positioning, ideal-client profiles, and voice rules don't depend on how many operating pillars the company has; they needed pointers to `17`, not rewrites.
- **`08_TECHNICAL_ARCHITECTURE.md`** — only the illustrative layout diagram in §5 changed. Stack decisions, rendering strategy, and the hybrid storage model are pillar-agnostic.
