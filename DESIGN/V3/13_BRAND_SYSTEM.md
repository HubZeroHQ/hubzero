# 13 — Brand System

> **Status: New foundational document — 2026-07-11.** This is the single most severe gap `CRITIQUE_01_REVIEW_BOARD.md` found in `DESIGN/V3`: every document from `00` to `10` argues, at length, that electric blue must be retired, and none of them addresses the one asset that's most visibly electric blue — the actual brand mark. This document resolves that gap. It governs the logomark, the wordmark, and every touchpoint outside the website itself where HubZero's identity appears — favicon, social cards, GitHub, package registries — so that shipping Working Blueprint's copper-now-signal-gold system never results in a warm, quiet, disciplined website sitting under a glossy blue 3D icon in the browser tab, which is the exact, avoidable embarrassment this document exists to prevent.

## 1. Brand invariants

Before any specific touchpoint is specified, these are the rules that hold across all of them — present and future, digital and (if it ever arises) physical, at any size from a 16px favicon to a full-bleed hero. Every section below is a specific application of these; none of them may be quietly relaxed by a future touchpoint's constraints without reopening this section explicitly.

1. **One mark, one geometry.** The logomark is a single, fixed shape. A favicon, a GitHub avatar, and a navbar icon are the same geometry rendered at different sizes — never different marks, never a "simplified for small sizes" alternate silhouette. If a context is too small for the real mark to read correctly, the fix is a larger minimum size (§3), never a second, simplified mark.
2. **Flat, never glossy or 3D-rendered.** The current asset (`public/brand/icon.png`, `icon.svg`, and the `primary`/`wordmark` renders) is a beveled, gradient-shaded, glossy 3D render — a rendering _style_, independent of and in addition to its electric-blue hue. Both are retired together. Working Blueprint's entire visual language is built from flat linework, hairline strokes, and real material color rather than simulated studio lighting (`02_VISUAL_LANGUAGE.md` §1, §6); a glossy 3D mark is exactly the kind of generic "tech logo" rendering convention (bevel, drop shadow, specular highlight) the rest of this folder spends ten documents arguing away from. No future version of the mark may reintroduce gloss, bevel, drop shadow, or a gradient fill, regardless of how the color story evolves.
3. **Monochrome-first, color-optional.** The mark is designed and specified as a single-ink flat shape first; signal gold (`11_COLOR_PHILOSOPHY_AMENDMENT.md` §4) is one valid rendering of it, not a requirement for it to be recognizable. This is what makes the mark actually survive a favicon, a GitHub org avatar rendered on GitHub's own dark or light chrome, a printed proposal document, or any context this folder doesn't anticipate today — a mark that only works in one exact hue on one exact background is not a mark, it's a screenshot.
4. **The Favicon Test.** A concrete, falsifiable gate, deliberately modeled on `01_VISION.md` §8's Recognition Test because that test's own weakness (per `CRITIQUE_01_REVIEW_BOARD.md` §32) is having no operational protocol — this one has one: **render the mark as a single flat color at 16×16px on both a light and a dark square. Does it still read as a distinct, intentional shape, not a smudge?** If not, the mark has failed, regardless of how well it reads at hero scale. This is the single cheapest, most-repeated rendering of any company's identity that exists (a browser tab, a bookmark, a search result favicon) and the one this folder's original set never once considered.
5. **No secondary, per-pillar marks.** Work, Labs, Builds, and Blueprints do not get their own logo variants or badges distinct from the one mark — the same discipline `11_COLOR_PHILOSOPHY_AMENDMENT.md` §6 already applies to color, applied here to the mark itself. Pillars differentiate through composition and register (`09_PAGE_ARCHETYPES.md`), never through a family of sub-logos.
6. **Never re-skinned onto a real artifact.** The mark is never applied as a color-overlay, duotone filter, or watermark onto a real photograph or diagram — the same artifact-integrity rule `02_VISUAL_LANGUAGE.md` §5 already applies to hardware photography and schematics, extended here to forbid the common social-card mistake of tinting a real screenshot to "match the brand."
7. **The wordmark's typography always matches the current type system, not a frozen legacy choice.** The mark's typography is not exempt from `03_TYPOGRAPHY.md` — when the type system changes (as it just did, §5 below), the wordmark changes with it, on the same timeline, not as a deferred afterthought.

## 2. The logomark: what's wrong, and the resolution

**The problem, stated plainly (`CRITIQUE_01_REVIEW_BOARD.md` Part 8, finding 1).** `public/brand/icon.png`, `icon.svg`, `primary.png`, `primary-horizontal.png`, and `wordmark.png` are all the same glossy, beveled, gradient-shaded electric-blue "H/Z" hexagonal monogram — a stylized H and a stylized Z/arrow sharing a hexagonal negative-space boundary, rendered with specular highlights and a drop shadow like a 2010s-era app icon. This asset is currently wired into `public/favicon.ico`, `apple-touch-icon.png`, both `android-chrome-*.png` sizes, `site.webmanifest` (whose `theme_color` is hardcoded `#3ABEFF` — the exact retired blue, baked directly into a manifest file no CSS token migration touches), and `src/components/brand/logo.tsx`'s navbar/footer lockup. Every other document in this folder argues to retire this exact hue and this exact "glossy tech" rendering register; none of them touched a single one of these files.

**The resolution is not a new mark from nothing — it's a re-execution of the existing geometry.** The current H/Z hexagonal monogram has real equity: it's a genuine typographic construction (an H and a Z sharing structure inside a hexagonal frame), not a generic stock symbol, and inventing a wholly new mark would be a bigger, less justified undertaking than this folder's actual mandate (retire an inherited color and rendering style, not the company's underlying visual DNA). **The fix: redraw the same H/Z monogram flat, in the site's own trace-path linework language** (`02_VISUAL_LANGUAGE.md` §1) — right angles and 45-degree diagonals only, no curves, no bevel, no gradient fill, rendered as either a solid flat fill or an open linework outline depending on context (§3). This makes the mark, for the first time, an actual instance of the same visual grammar the rest of the site uses, rather than a legacy asset from a different design era sitting on top of it — directly resolving the critique's finding that "a future team implementing this blueprint would ship a copper-and-graphite identity with a glossy blue 3D logo sitting in the navbar."

**Construction logic:**

- The monogram's existing hexagonal boundary and H/Z letterform logic are retained — this is the recognizable, real geometry worth keeping.
- Every edge that is currently a beveled 3D facet becomes a single flat plane bounded by a real trace-path line (right angle or 45°, per `02_VISUAL_LANGUAGE.md` §1) — the mark should look like it was drawn on the same drafting sheet as everything else on the site, not rendered in a 3D application.
- The mark ships in exactly two treatments: **solid** (a single flat fill, signal gold or ink, for favicon/small-size use where linework detail would disappear) and **linework** (an open, hairline-stroked outline in the same weight as the site's diagram linework, for large-scale use — a hero moment, a GitHub social preview image, a proposal document's header) — never the gradient-shaded "in between" the current asset uses.
- No drop shadow, ever, under either treatment — consistent with `04_COLOR.md` §6–7's rejection of glow and colored shadows, extended here to the mark itself.

## 3. Minimum size and clear space

- **Minimum digital size: 16px** (the favicon case, §1 test 4) for the solid treatment; the linework treatment requires meaningfully more room to read its internal detail and should not be used below roughly 32px.
- **Clear space:** a minimum margin equal to the mark's own cap-height (the height of the H stroke) on all four sides, in any lockup — the same "unpopulated substrate" logic `02_VISUAL_LANGUAGE.md` §1 already applies to page composition, applied here at the mark's own scale so it's never crowded by adjacent nav items, a page title, or a browser chrome element.
- **Never scaled anisotropically** (stretched wider or taller than its native proportions) and never rotated for a "dynamic" placement — the mark's geometry is fixed, per invariant 1.

## 4. Color application

Per `11_COLOR_PHILOSOPHY_AMENDMENT.md` §7: the mark's chromatic version uses signal gold, justified purely as the studio's one functional signal color applied to its own mark — not as a claim that the mark itself "is" hardware, software, or AI. Concretely:

- **On dark (solder-mask) surfaces:** signal gold solid fill, or ink-on-transparent linework where a quieter treatment is wanted (a footer, a document footer).
- **On light (drafting-paper) surfaces:** ink solid fill (never signal gold as the default light-mode fill — gold-on-paper reads closer to a caution/warning register than a brand mark at most real-world contrast levels; verify this specifically under `12_ACCESSIBILITY.md` §1's contrast requirements before finalizing) or gold linework as a considered secondary option.
- **Single-color/monochrome contexts** (a favicon rendered on an OS that ignores transparency, a black-and-white print document, a GitHub org avatar on an unpredictable background): pure ink or pure white flat fill, no gold, no gradient — this is what invariant 3 exists to guarantee always works.
- **Never the retired electric blue, in any context, including cached or legacy files** — this explicitly includes `site.webmanifest`'s `theme_color` and `background_color` fields, `apple-touch-icon.png`, both `android-chrome-*.png` sizes, and any hardcoded hex reference to `#3ABEFF` outside the CSS token system, all of which currently carry the retired color and are easy to miss because they sit outside `globals.css`'s token architecture entirely.

## 5. The wordmark

**The current inconsistency:** `src/components/brand/logo.tsx` sets "Hub" in Geist Sans and "Zero" in Instrument Serif italic — a real, deliberate typographic choice at the time it was built, and the single most direct tie between the nav-level brand lockup and the site's editorial typographic identity. `03_TYPOGRAPHY.md` §6 retires Instrument Serif in favor of IBM Plex Serif, used upright by default (§3 of that document) rather than italic-by-default — a decision made for the site's body and display type that the wordmark, sitting in the same codebase, was never revisited against.

**Resolution:** the wordmark migrates on the same timeline as the rest of the type system (`10_IMPLEMENTATION_ROADMAP.md` Phase 1, §7 below) — "Hub" stays Geist Sans (Semibold, matching `03_TYPOGRAPHY.md` §2's display weight ceiling), "Zero" moves to IBM Plex Serif, upright, matching the rest of the site's post-migration Plex Serif moments rather than retaining italic as a frozen exception. This is not a new design decision — it's applying a decision `03_TYPOGRAPHY.md` already made to the one place in the codebase that decision hasn't reached yet.

**Lockup rules:**

- **Full lockup** (icon + wordmark): the icon's linework treatment (§2) at a size matched to the wordmark's cap-height, wordmark immediately following at the clear-space margin (§3) — the existing `Logo` component's `variant="full"` structure is correct and is retained; only the icon asset and the serif family change.
- **Icon-only** (`variant="icon"`, mobile nav, favicon-adjacent UI): the solid treatment (§2) at or above the 16px minimum (§3).
- **Never a stacked (icon-above-wordmark) lockup** for the primary brand treatment — HubZero's nav and footer usage is exclusively horizontal, and introducing a second lockup geometry would violate invariant 1's "one mark" spirit extended to the full lockup as a unit.

## 7. Social cards (Open Graph / Twitter cards)

Per-page-type dynamic generation, not one static image — a single generic "HubZero" card reused across every case study, Note, and Labs entry fails the same "real evidence over generic decoration" test `02_VISUAL_LANGUAGE.md` and `07_IMAGERY.md` already apply to on-page imagery, extended here to the one image most likely to be seen by someone who's never visited the site (a shared link's preview card).

- **Default/homepage card:** the linework-treatment mark (§2) at generous scale on a solder-mask-dark field, with the wordmark (§5) — quiet, and correctly representing the site's own material palette rather than a generic "brand card" template.
- **Case study / Notes / Labs / Blueprints cards:** generated per entry (a Next.js `opengraph-image` route, computed from real document data — title, client or category, and, where one exists, the entry's real cover image cropped to the card's aspect ratio) rather than a static fallback for every entry. This is the same "real artifact over decoration" discipline `07_IMAGERY.md` §1 already requires on-page, applied to the one surface most likely to be judged before a visitor ever reaches the actual page.
- **Typography on generated cards** follows `03_TYPOGRAPHY.md`'s technical-label register (Geist Mono for the category/date line) plus a Geist Sans headline — never the wordmark's serif treatment shrunk down illegibly, and never a generic drop-shadowed text-on-photo treatment.
- **Never fabricated content:** a card for an entry with no real cover image shows the default card's quiet treatment (§7's first bullet) rather than a generic stock image or an AI-generated "concept" background — the same zero-fabrication standard `07_IMAGERY.md` §1 applies everywhere else, extended to the one surface a design system most commonly forgets to hold to it.

## 8. GitHub

`github.com/HubZeroHQ` is a real, existing touchpoint this folder's original ten documents never mentioned:

- **Org avatar:** the solid-treatment mark (§2, §4), monochrome-safe per invariant 3, since GitHub renders org avatars on both its light and dark theme without the org controlling the background.
- **Repository social preview image** (the image GitHub generates for link previews and that a maintainer can override in repo settings): the same default card treatment as §7's homepage card, sized to GitHub's specific preview-image convention.
- **Repository README header:** if a README ever carries a header image (the current `README.md` does not, and should not gain one purely for decoration) — no requirement to add one; if one is ever added, it follows the same mark-and-wordmark lockup rules as §5, never a separate "GitHub-specific" logo treatment.
- **No separate "GitHub brand"** — invariant 5 applies here as much as it does to product pillars; a GitHub-specific badge or mascot would be exactly the kind of secondary mark this document exists to prevent.

## 9. Package registries

HubZero's `package.json` (`"private": true`) is not currently published to npm or any other registry, and this document does not fabricate a publishing story that doesn't exist yet — per the same zero-fabrication standard applied everywhere else in this folder. What this document commits to, for whenever that changes (a Blueprint distilled into a real, published package, per `ARCHITECTURE/17_COMPANY_STRUCTURE.md` §2's own description of what a Blueprint is):

- **A published package's README** follows the same wordmark and mark rules as GitHub (§8) at the top of the README, plus real, honest metadata (`author`, `homepage`, `repository` fields pointing at real, live URLs) — never a placeholder npm badge count or a fabricated "downloads" claim before real usage exists.
- **No registry-specific visual treatment** beyond what §5–8 already specify — a package README is a GitHub-adjacent surface, not a fourth brand context requiring its own rules.

## 10. Other non-website touchpoints

Named honestly rather than speculatively over-specified, per this folder's own standing discipline (`01_VISION.md` §10) that an honest gap beats an invented one:

- **Proposal and contract documents** (a real, likely-existing touchpoint for a studio closing six-figure engagements) should use the linework mark and wordmark lockup (§2, §5) as a document header, in ink-on-paper (never signal gold as body-document chrome) — flagged here as a real, near-term need worth a small follow-up production pass, not specified further because no such document template exists in this repository to audit against.
- **Email signatures, pitch decks, and any other physical/document touchpoint** not enumerated above inherit the invariants in §1 by default (one mark, flat, monochrome-safe) until a dedicated pass addresses them specifically — this document deliberately does not invent detailed rules for touchpoints nobody has built yet.

## 11. Where this fits in the roadmap

This document's asset work — redrawing the mark (§2), rebuilding the favicon/app-icon set (§6), and migrating the wordmark's serif (§5) — belongs in `10_IMPLEMENTATION_ROADMAP.md` **Phase 1**, alongside the color and type token migration, not deferred to a later phase. The roadmap's own sequencing principle 3 ("never ship a partial migration that leaves two competing systems live simultaneously") applies with unusual force here: shipping new copper-turned-signal-gold tokens and new typography while the navbar still renders the glossy electric-blue 3D mark is the single most visible, most avoidable form of exactly that failure, and it is the specific outcome `CRITIQUE_01_REVIEW_BOARD.md` named as the most likely embarrassing result of implementing this folder's original ten documents literally. Phase 1's effort estimate in `10_IMPLEMENTATION_ROADMAP.md` §3 is amended to include this asset work explicitly, not as an unscoped addition discovered mid-phase.
