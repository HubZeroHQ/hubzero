# 15 — Diagram System

> **Status: New foundational document — 2026-07-11.** `02_VISUAL_LANGUAGE.md` §9 and `06_COMPONENT_LANGUAGE.md` §15 already establish diagrams as "the single most important visual device in this identity" and specify their visual convention in detail. Neither document addresses the question a real implementation hits immediately: **who authors a given diagram, and through which system?** `ARCHITECTURE/20_CONTENT_BLOCKS.md` already ships a fifteen-block CMS content model that non-designer teammates use to author case studies, Labs entries, and Notes — diagrams have to live somewhere inside, or deliberately outside, that model, and nothing in `DESIGN/V3` says which. This document draws that line.

## 1. The two kinds of diagram, and why they can't be the same system

**Editorial diagrams** are the ones a teammate authors while writing a case study, a Labs entry, or a Note — a simple system sketch accompanying a technical write-up, a before/after comparison, a small process flow. They need to be **CMS-editable**: created and revised by whoever is writing the content, without a code change, the same way every other block in `ARCHITECTURE/20_CONTENT_BLOCKS.md`'s model is.

**Engineering diagrams** are the ones that depict HubZero's own real, load-bearing systems at full fidelity — the Labs/R&D data-flow diagram `DESIGN/00_AI_DESIGN_GUIDE.md` §2 already cites as the reference case, a case study's real architecture walkthrough (`09_PAGE_ARCHETYPES.md` §3, §5), a Blueprint's own structure diagram. These are **hand-built as real React/SVG components** (`06_COMPONENT_LANGUAGE.md` §15's "built from the site's real type and line tokens" instruction), because they need capabilities a generic content-authoring tool cannot honestly provide: GSAP-sequenced build-order animation (`08_MOTION_SYSTEM.md` §3), pixel-precise trace-path geometry that must line up exactly with real system boundaries, and a level of visual craft that is, correctly, an engineering task rather than a content-authoring one.

**Why one system can't serve both jobs, stated plainly:** a CMS-editable diagram builder capable of the geometric precision and animation `06_COMPONENT_LANGUAGE.md` §15 demands would, in practice, be a small diagramming application in its own right — a real, ongoing engineering investment `ARCHITECTURE/19_CMS_FOUNDATION.md`'s generic CRUD engine was never built to be, and one this folder's own `10_IMPLEMENTATION_ROADMAP.md` §11 explicitly warns against scoping in ("no new CMS collections, no new content-block types... any future team tempted to add new backend scope alongside it should treat that as a separate, unjustified expansion"). Conversely, forcing every editorial diagram through the full engineering-diagram component would mean a teammate writing a Note can't add a simple illustrative sketch without a code change — exactly the kind of bottleneck `ARCHITECTURE/20_CONTENT_BLOCKS.md` §0 was built to remove from narrative authoring generally. The right answer is two tiers with a clear, statable rule for which one a given diagram belongs to, not one system stretched to cover both jobs badly.

## 2. The deciding question

Before authoring any diagram, ask: **does this depict a real system precisely enough that its exact shape is part of the evidence, or does it illustrate a relationship for a reader's understanding?** A signal-path diagram of an actual board, a case study's actual service architecture, a Blueprint's actual module structure — the diagram's precision _is_ the content; a wrong box or a mislabeled arrow would be a factual error, not a stylistic one. That's an engineering diagram (§4). A simple flow sketch in a Note explaining a concept in general terms, a Labs entry's rough "here's roughly how the pieces fit together" sketch, a before/after comparison — these illustrate rather than evidence, and precision to the pixel doesn't change their communicative job. That's an editorial diagram (§3).

**A genuine gray area exists and should be resolved toward the engineering tier, not the editorial one, whenever real accuracy is being claimed.** `01_VISION.md` §10's zero-fabrication standard, extended to visuals, means a diagram that looks precise enough to be read as a real technical drawing but was authored in a simplified editorial tool is a quiet accuracy risk — if a reader would reasonably take the diagram as an accurate representation of a real system, it needs the engineering tier's actual rigor, not just its visual style.

## 3. Editorial diagrams — the CMS-authored tier

**Mechanism: a new block type, `diagram`, added to `ARCHITECTURE/20_CONTENT_BLOCKS.md`'s existing fifteen-type vocabulary as a sixteenth — the first addition to that closed list since it shipped, and one that clears the bar §1.1 of that document sets for any addition (a real, recurring editorial need already identified across multiple collections, not a speculative one).**

- **What it captures, structurally, not visually:** a small set of labeled nodes and the connections between them — `{ nodes: [{ id, label }], connections: [{ from, to, label? }] }` — deliberately data, not freeform drawing. The author places nodes and draws connections between them through a constrained editor (drag-to-position within a grid-snapped canvas, matching the site's own 12-column/spacing-scale grid, `14_VISUAL_TOKENS.md` §1–2), never a blank-canvas freehand tool.
- **Rendering is automatic, not authored:** the block's stored data (nodes and connections) is rendered through the site's own diagram-rendering primitive — the same trace-path geometry, drafting-cyan linework, and type tokens every diagram on the site uses (`02_VISUAL_LANGUAGE.md` §9, `14_VISUAL_TOKENS.md` §4, §6) — so an editorially-authored diagram is visually indistinguishable in _style_ from an engineering one, and differs only in the complexity and animation it supports. An author picks _what_ to connect to _what_; the system, not the author, decides exactly how a right-angle trace routes between two nodes, exactly which hairline weight it uses, and exactly which cyan value it renders in — this is what keeps a non-designer-authored diagram from becoming the "five different engineers inventing five different diagonal-line styles" failure `CRITIQUE_01_REVIEW_BOARD.md` §39 named for icons, applied here to the exact same risk in diagrams.
- **Explicitly out of scope for this block, matching the discipline `ARCHITECTURE/20_CONTENT_BLOCKS.md` §3 already applies to `accentImage`/`difficulty`:** GSAP build-order sequencing, pixel-precise custom node shapes (a node is always the same small square/diamond primitive, never a custom schematic symbol), and more than roughly eight to ten nodes on one diagram — a `diagram` block that needs more complexity than that has, in practice, become an engineering diagram (§2's deciding question) and should be built as one instead, not pushed further into a tool that was never meant to carry that weight.
- **Validation follows the exact pattern every other block type already uses** (`ARCHITECTURE/20_CONTENT_BLOCKS.md` §1, §2): a Zod schema in the same `discriminatedUnion("type", …)` every block's `data` already validates against, no second validation mechanism, no special-cased save path. `getMediaUsage()`-style reference scanning doesn't apply here (a `diagram` block references no `Media` documents), but the same "the CMS's generic engine gets one narrow, well-typed addition, not a parallel system" posture `ARCHITECTURE/20_CONTENT_BLOCKS.md` §2 already states for the `blocks` field type applies identically to adding `diagram` as a sixteenth block type.
- **Where it appears:** anywhere the existing `content: Block[]` field already lives — Case Studies, Labs Projects, Builds, Blueprints, Notes — with no new collection and no new route, consistent with `10_IMPLEMENTATION_ROADMAP.md` §11's standing constraint.

## 4. Engineering diagrams — the code-authored tier

**Mechanism: unchanged from `06_COMPONENT_LANGUAGE.md` §15** — a real React/SVG component, hand-built per system by an engineer, using the site's real type and line tokens directly rather than through any authoring UI. This tier is not new; this document's job is to state clearly that it is a _separate_ system from §3, not a more-complex mode of it, and to specify how the two coexist visually and architecturally.

- **What it can do that §3 can't:** real schematic/box-arrow/queue convention appropriate to its specific domain (`02_VISUAL_LANGUAGE.md` §9), GSAP-sequenced build-order animation (`08_MOTION_SYSTEM.md` §3), arbitrary node counts and connection complexity, and pixel-exact correspondence to a real system's actual shape.
- **Where it appears:** embedded directly in a page's markup at the specific point in a narrative that warrants it (a case study's "how it's built" section, a Services/Hardware practice page's signal-path diagram, a Labs entry's current architecture) — not authored through the CMS, because these diagrams are, correctly, an engineering artifact in the same sense a real code sample is: authored by whoever built the system it depicts, at the same rigor as the system itself.
- **A `content` block can still reference one:** where an engineering diagram needs to appear inside an otherwise CMS-authored narrative (a case study's body text flowing around a real architecture diagram), the existing `code` or `html` block types (`ARCHITECTURE/20_CONTENT_BLOCKS.md` §1.1) are the correct integration point for an engineer to embed a reference to a named, hand-built diagram component — never a reason to expand the `diagram` block (§3) to somehow also support this tier's complexity. This mirrors exactly how `07_IMAGERY.md` and `06_COMPONENT_LANGUAGE.md` §14 already treat real code: authored by an engineer, placed by an editor, rendered through the CMS's existing generic block renderer.

## 5. Visual coherence across both tiers

Both tiers share the same tokens (`02_VISUAL_LANGUAGE.md` §9–11, `14_VISUAL_TOKENS.md` §4, §6) so a reader never perceives a "CMS diagram" register as visually distinct from an "engineering diagram" register — the Recognition Test (`01_VISION.md` §8) has to hold across both, and a visitor should never be able to tell, just by looking, whether a given diagram came from a teammate's Note or an engineer's page build. What differs between the tiers is authoring mechanism and complexity ceiling, never visual language, color, or type.

## 6. Accessibility and provenance

Both tiers inherit `12_ACCESSIBILITY.md` §3 in full — a real text alternative, legibility with color removed, and keyboard/screen-reader reachability for any interactive annotation — with one addition specific to the editorial tier: **a `diagram` block's node labels are the diagram's own accessible-name source** (each node's `label` field doubles as its accessible name in the rendered output), so an author who fills in real, descriptive node labels while building the diagram has, by construction, already done the accessibility work — a deliberate design choice so this requirement doesn't depend on a non-designer author remembering a separate accessibility step.

Per `01_VISION.md` §10 and `02_VISUAL_LANGUAGE.md` §9's "depict something that actually exists or actually happened" rule, both tiers are held to the same non-fabrication standard: an editorial `diagram` block sketching a real, simplified relationship is honest; one that invents a system relationship that doesn't exist to make an entry look more developed is a fabrication exactly as serious as an invented metric, regardless of which authoring tool produced it.

## 7. What this means for the roadmap

The `diagram` block type (§3) is new implementation scope this document adds to `ARCHITECTURE/20_CONTENT_BLOCKS.md`'s otherwise-closed model — it belongs in `10_IMPLEMENTATION_ROADMAP.md` **Phase 7** (full diagram and imagery system rollout), alongside the reusable engineering-diagram component that phase already scopes, since both draw from the same underlying rendering primitive (§3's "rendering is automatic" point) and are naturally built together rather than sequentially. This is an addition to Phase 7's scope, not a new phase — Phase 7 was already the ongoing-production home for "build out the reusable architecture-diagram component," and a CMS-side sibling to that same component is the same workstream, not a separate one.
