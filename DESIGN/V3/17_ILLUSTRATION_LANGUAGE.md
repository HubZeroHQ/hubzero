# 17 — Illustration Language

> **Status: New foundational document — 2026-07-11. Additive only.** This document modifies no existing document and alters no existing principle. `02_VISUAL_LANGUAGE.md` §8–9 and `07_IMAGERY.md` §1 already established that "illustration," in HubZero's vocabulary, means a real diagram drawn well — never a metaphor standing in for a claim with no real referent — and this document does not loosen that standard anywhere. What it adds: a named, first-class illustration system that sits alongside photography (`07_IMAGERY.md`) and documentary diagrams (`02_VISUAL_LANGUAGE.md` §9–11, `15_DIAGRAM_SYSTEM.md`) as a third, equally disciplined register — grown directly from the HZ monogram's own construction logic (`13_BRAND_SYSTEM.md` §2) — for the real contexts (hero sections, social cards, presentations) that today have no honest visual option at all when no specific real system exists yet to diagram.

## 1. What illustration means here, and its boundary with existing categories

`07_IMAGERY.md` §1 holds that every image sits on one of two poles — a real artifact, or a site-native diagram depicting a real system's structure — and forbids anything in between. This document does not add a third pole to that spectrum. It names and extends the formal vocabulary the site-native pole already implies (`02_VISUAL_LANGUAGE.md` §9's "real, correct technical convention for whatever it depicts," `07_IMAGERY.md` §5's exploded-assembly and orthographic CAD conventions) into a broader, dimensional range — faceted volumetric forms, isometric projection, exploded assemblies — and gives it a name, a construction discipline, and a set of contexts to live in that the original two-pole framing never addressed, because those contexts were never about documenting one specific real system in the first place.

**Two registers, not two new systems** — this document reuses the exact canonical/expressive duality `13_BRAND_SYSTEM.md` §1 invariant 2 already established for the mark, because it's the same underlying distinction:

- **Documentary illustration.** Flat, schematic, Ice Blue linework (`11_COLOR_PHILOSOPHY_AMENDMENT.md` §6), depicting a specific real system that actually exists or actually happened. This is exactly `02_VISUAL_LANGUAGE.md` §9's existing diagram category and `15_DIAGRAM_SYSTEM.md`'s two authoring tiers — nothing here changes; documentary illustration _is_ a diagram, by another name, and every existing rule governing diagrams governs it in full.
- **Constructional illustration** (the one genuinely new category this document adds). Dimensional, faceted, isometric or exploded-assembly-derived work, in the spirit of the mark's own expressive geometry, that depicts a real, named technical category or practice — how a multilayer PCB is actually constructed, how a signal traverses a typical embedded system, what an exploded enclosure assembly actually looks like — without claiming to document one specific real HubZero deliverable. §2 states the honesty test that keeps this category from becoming a loophole.

## 2. The Textbook Test — the honesty boundary for constructional illustration

`01_VISION.md` §10 is unchanged and unambiguous: illustration is never authorized as a substitute for a real diagram or photograph HubZero doesn't have yet — an honest gap beats a commissioned stand-in, always. Constructional illustration does not touch that rule, because it is never used to fill an evidence gap. It exists only for contexts that were never evidentiary to begin with — a homepage hero, a social card, a presentation cover — where the honest options today are text alone or nothing (`07_IMAGERY.md` §1's existing fallback), and where a real, disciplined illustration is a better answer than either.

A constructional illustration must pass **the Textbook Test**, deliberately modeled on this folder's other concrete, falsifiable gates (the Recognition Test, the Uniqueness Test, the Favicon Test): **would a real engineering textbook or trade publication draw this the same way, for the same honest, general-explanatory reason?** Concretely, three conditions, all required:

1. **Real, correct convention.** It uses actual isometric projection rules, actual exploded-assembly logic, actual signal-routing geometry — the same standard `02_VISUAL_LANGUAGE.md` §9 already applies to documentary diagrams. An invented pseudo-technical convention fails this immediately.
2. **No false specific claim.** It is never captioned, placed, or presented in a way that implies it documents one specific real HubZero client, product, or metric it doesn't. If a reader could reasonably mistake it for evidence of a specific real thing, it has failed — the fix is either to build the real documentary diagram (if the real thing exists) or to remove the illustration, never to leave the ambiguity standing.
3. **A real, named general category.** It depicts something a real engineer would recognize as an accurate instance of a real practice or structure — not an arbitrary visual metaphor. This is the same distinction `02_VISUAL_LANGUAGE.md` §8 already draws between a real diagram and "a rocket for growth, a puzzle piece for integration" — a constructional illustration is closer to a textbook's generic transistor cross-section than to either.

## 3. Formal vocabulary

Every device in this vocabulary is a specific, named extension of something an existing document already sanctions — nothing here is invented from nothing:

- **Faceted forms and precision cuts** — the direct formal descendant of the HZ monogram's own expressive construction (`13_BRAND_SYSTEM.md` §2's flat-plane-bounded-by-trace-path-line logic, extended from the mark's fixed geometry to open subject matter).
- **Engineering construction lines, dimension and leader lines** — `02_VISUAL_LANGUAGE.md` §10's existing annotation vocabulary, unchanged, reused here as illustration content rather than only diagram overlay.
- **Exploded assemblies, isometric views, volumetric geometry** — real CAD convention `07_IMAGERY.md` §5 already sanctions for actual enclosures HubZero has designed, extended here to real, general assembly conventions that aren't tied to one specific enclosure.
- **Signal paths** — `02_VISUAL_LANGUAGE.md` §1's trace-path geometry (right angles and 45-degree diagonals only), the same signature line quality governing every other connective device on the site.
- **System diagrams and wireframes** — `02_VISUAL_LANGUAGE.md` §9, `06_COMPONENT_LANGUAGE.md` §15, `15_DIAGRAM_SYSTEM.md`'s existing diagram categories, unchanged.
- **Technical annotation** — `02_VISUAL_LANGUAGE.md` §10's leader-line and call-out convention, set in `03_TYPOGRAPHY.md` §7's technical-label register (Geist Mono).

## 4. Construction discipline

- **Documentary illustration** stays within `14_VISUAL_TOKENS.md` §4's diagram-linework stroke weight (1.5px) and the flat, right-angle/45°-only geometry `02_VISUAL_LANGUAGE.md` §1 governs everywhere else — no exceptions, since it is a diagram.
- **Constructional illustration** may use real dimensional facets and light behavior, per `13_BRAND_SYSTEM.md` §2's expressive-register logic extended from the mark to open subject matter — but never gloss, bevel, or lighting invented for its own sake; every facet and highlight should be traceable to the same real-light logic `04_COLOR.md` §6 already requires of the mark and the site generally (real light behavior, never an ambient glow with no referent).
- **Both registers draw exclusively from `11_COLOR_PHILOSOPHY_AMENDMENT.md` §2's seven-stop palette.** No illustration, in either register, introduces a hue outside that ramp — the same "no third signal color" discipline `04_COLOR.md` §8 and `11_COLOR_PHILOSOPHY_AMENDMENT.md` §8 already hold the rest of the system to.
- **Icons (`14_VISUAL_TOKENS.md` §6) are this vocabulary's smallest unit** — the same 24×24 grid, 1.5px stroke, right-angle/45°-only construction rule governs an icon and a documentary illustration identically, because an icon is, formally, the smallest possible documentary illustration.

## 6. Where photography remains correct

Unchanged in every particular, per `07_IMAGERY.md`: team and founder photography, workspace and office photography, real hardware (PCB renders, macro photography, prototypes), and any real, specific client project or deliverable. If a reader could reasonably ask "is this real, and is it this specific thing?", the only honest answers are a real photograph or a real documentary diagram of that specific system — never a constructional illustration, which by definition (§2, test 2) never makes that claim. `07_IMAGERY.md` §14's placement table governs all of this without modification.

## 7. Coexistence — one visual language, not two

Illustration and photography are not two competing visual languages sharing a site; they are one visual language covering two different jobs — proving what's specific and real (photography, documentary diagrams) and expressing what's general and true (constructional illustration) — the same distinction `01_VISION.md` §7's "editorial engineering" philosophy already draws between evidence and editorial framing, extended here to say illustration is part of the framing, never part of the evidence.

- **Shared governing constraints.** Both draw from the same palette (`11_COLOR_PHILOSOPHY_AMENDMENT.md` §2), the same trace-path/right-angle geometry where flat, and the same annotation and technical-label register (`02_VISUAL_LANGUAGE.md` §10, `03_TYPOGRAPHY.md` §7) — a reader should never be able to tell, from construction alone, that an illustration and a diagram came from different systems.
- **Never composited together.** An illustration is never overlaid onto, or blended directly into, a real photograph — the same artifact-integrity rule `02_VISUAL_LANGUAGE.md` §5 and `13_BRAND_SYSTEM.md` invariant 6 already apply to the mark and to real photography, extended explicitly here to forbid the common generic-agency trick of illustrating decorative elements onto a real product photo.
- **Sequencing on a page.** Illustration opens and frames — a title-block moment (`02_VISUAL_LANGUAGE.md` §1), a section divider, a hero — while photography and documentary diagrams prove. A page that reverses this (illustration used at the evidence beat, photography used as a framing device) has confused the two jobs.

## 8. Anti-patterns — a rejected-alternatives gallery

Concrete rejections, most already excluded elsewhere and restated here so this vocabulary isn't mistaken for license to reintroduce them:

- **Generic SaaS gradient blobs, abstract gradient shapes** — already excluded by `DESIGN/00_AI_DESIGN_GUIDE.md` §3.2 and the gradient discipline in `04_COLOR.md` §5 / `11_COLOR_PHILOSOPHY_AMENDMENT.md`; illustration inherits this exclusion in full. _This, not that:_ a faceted, isometric exploded circuit-board assembly drawn in the mark's own construction logic — not a glowing gradient orb with abstract floating particles.
- **Cartoons, mascots, 3D character illustration** — already excluded by `DESIGN/00_AI_DESIGN_GUIDE.md` §3.3 and `07_IMAGERY.md` §15's closed list; `13_BRAND_SYSTEM.md` invariant 5's ban on per-pillar mascots extends the same logic to illustration generally. _This, not that:_ a Labs entry's concept illustration showing a real signal-path convention — not a cartoon robot or a generic "AI brain" graphic.
- **Cyberpunk and neon-glow effects** — not previously named in this folder, rejected here on the same "timeless, not trendy" standard `01_VISION.md` and `03_TYPOGRAPHY.md` §6 already apply to type and color: a dateable aesthetic register tied to a specific, passing visual moment, and in direct tension with `01_VISION.md` §2's "unhurried, precise" emotional register.
- **AI-generated "concept" tech art** — already excluded by `01_VISION.md` §10 and `07_IMAGERY.md` §1's closed list; restated here specifically because constructional illustration's real-category allowance (§2) could otherwise be mistaken for permission to do exactly this. It is not — every constructional illustration must pass the Textbook Test's real-convention requirement, which a generative "concept art" prompt does not.

## 9. What this document does not do

It does not create a new CMS collection or block type. A constructional illustration is a designed graphic asset (an authored SVG or a rendered image), stored and placed exactly like any other real image — through the existing `image`/`gallery` block types and the `Media` collection (`ARCHITECTURE/20_CONTENT_BLOCKS.md` §1.1) — never through `15_DIAGRAM_SYSTEM.md` §3's `diagram` block, which is reserved for structured node/connection data and governs documentary illustration's CMS-authored cases only. This keeps `10_IMPLEMENTATION_ROADMAP.md` §11's standing constraint (no new CMS collections or block types beyond the one already sanctioned) intact.

It does not specify exact SVG construction values, exact facet counts, or a finished illustration set — that is production work, paced the same way `10_IMPLEMENTATION_ROADMAP.md`'s Phase 7 already paces real photography and diagram-component production: ongoing, scaling with real content, not a single phase with a completion date.
