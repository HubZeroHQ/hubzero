# 14 — Visual Tokens

> **Status: New foundational document — 2026-07-11.** `CRITIQUE_01_REVIEW_BOARD.md` Part 7 ranked this the single highest-probability source of real, incompatible implementation divergence in the entire folder: `05_LAYOUT_SYSTEM.md` describes a spacing scale with no starting value and no step count, a grid with no stated column count, and `02_VISUAL_LANGUAGE.md` §12 describes an icon language with zero actual icons specified. This document gives both the concrete numbers the critique found missing — still one level above final hex/pixel tokens ready to paste into `globals.css` (`10_IMPLEMENTATION_ROADMAP.md` Phase 1 still owns final calibration against real screens), but concrete enough that two different engineers building from this document independently should converge, not diverge.

## 1. Base unit and spacing scale

`05_LAYOUT_SYSTEM.md` §2 describes "a modest geometric/near-Fibonacci progression (each step roughly 1.5–1.6× the previous)" with no starting value or step count. This document commits to both:

- **Base unit: 4px** — unchanged from `ARCHITECTURE/07_DESIGN_SYSTEM.md` §1's existing typographic base unit, kept as the shared root for spacing too, so type and space are provably drawn from one system rather than two coincidentally-similar ones.
- **The scale, nine steps, each roughly 1.55× the previous, rounded to the nearest 4px:**

| Step | Value | Typical use                                                                                                                                                  |
| ---- | ----- | ------------------------------------------------------------------------------------------------------------------------------------------------------------ |
| 0    | 4px   | Icon-to-label gaps, tightest technical-label tracking contexts                                                                                               |
| 1    | 8px   | Inline element gaps, tag/chip internal padding                                                                                                               |
| 2    | 12px  | Compact component internal padding                                                                                                                           |
| 3    | 20px  | Standard component internal padding, gap between a label and its value                                                                                       |
| 4    | 32px  | Gap between related elements within one beat (a heading and its immediate body copy)                                                                         |
| 5    | 48px  | Gap between distinct elements within one section (a metrics row's internal divisions)                                                                        |
| 6    | 76px  | Compact section vertical padding (a credibility strip, a step list — the "quiet beat" floor from `05_LAYOUT_SYSTEM.md` §2)                                   |
| 7    | 120px | Standard section vertical padding (the 96–128px floor `05_LAYOUT_SYSTEM.md` §2 already sets, now anchored to a named step rather than a free-floating range) |
| 8    | 188px | Tallest-beat vertical padding (a Featured Case Study, a Blueprint's live-demo hero — `05_LAYOUT_SYSTEM.md` §3's "primary dimension")                         |

**How the dimension-chain rule (`02_VISUAL_LANGUAGE.md` §3, `05_LAYOUT_SYSTEM.md` §3) uses this table concretely:** a page's tallest beat is assigned step 8; every other beat's padding is chosen from steps 5–7 in proportion to its real narrative weight, never a value outside this table, and never the same free-hand "whatever looks right" value repeated with a slight variation. A future engineer facing a spacing decision this table doesn't obviously answer should round to the nearest step rather than inventing an intermediate value — the whole point of a named scale is that "in between" values don't exist.

## 2. Grid

`05_LAYOUT_SYSTEM.md` §2 describes grid columns only through an example ("an 8/4 or 7/5 asymmetric split") without stating the base column count they're splits _of_. This document commits to:

- **12-column base grid** at the Marketing and Drafting width tiers (`05_LAYOUT_SYSTEM.md` §1) — a real, industry-standard column count chosen specifically because it divides cleanly into the asymmetric splits `05_LAYOUT_SYSTEM.md` already names as examples (8/4, 7/5, and additionally 9/3, 5/7) without forcing an odd, non-repeatable fraction for any of them.
- **Prose tier (`05_LAYOUT_SYSTEM.md` §1) is not gridded** — it is a single content column at its stated 720–760px width; a 12-column grid serving long-form reading content would be a false precision with no real use, since prose has no columns to split.
- **Gutters: 32px at Marketing width, 40px at Drafting width** (both drawn from the spacing scale in §1 — step 4 and a value between steps 4–5 respectively), wide enough to read as genuine "unpopulated substrate" (`05_LAYOUT_SYSTEM.md` §2) rather than a cramped seam, per that document's own stated requirement.
- **Column spans are always whole numbers of the 12-column grid** — an asymmetric split is expressed as `8/4`, `7/5`, `9/3`, or `5/7` (never, for example, a `7.5/4.5` split), and any layout that seems to need a non-whole-number split is a signal the content should be re-scoped to fit a real split, not that the grid should flex to accommodate it.

## 3. Corner radii

Not addressed anywhere in `02`–`10` at a token level, despite `06_COMPONENT_LANGUAGE.md` §1 explicitly specifying "a small, consistent corner radius" for buttons without a number, and §7 moving tags to "hairline-bordered rectangles" without specifying whether that rectangle has any radius at all.

- **4px** — the one radius value used sitewide, on buttons, form inputs, and any other rectangle that needs its corner softened at all. A single value, not a scale, because Working Blueprint's rectangle-not-pill instinct (`06_COMPONENT_LANGUAGE.md` §1) is explicitly about reading as "a labeled control on an instrument panel," and an instrument panel's controls don't have a hierarchy of corner roundness — they have one consistent manufacturing tolerance.
- **0px (square corners)** on tags (`06_COMPONENT_LANGUAGE.md` §7), table cells (`06_COMPONENT_LANGUAGE.md` §9), technical-label chips, and any element meant to read as a real schematic component designator (`R1`, `U3`) rather than a UI affordance — the distinction between a 4px-radius button and a 0px-radius tag is itself meaningful (button = interactive control, tag = data label), and collapsing that distinction to one universal radius would remove a real, useful signal.
- **No radius above 4px, anywhere** — a larger radius reads as a consumer-app convention (`06_COMPONENT_LANGUAGE.md` §1's own reasoning against the pill shape, generalized to any oversized radius).

## 4. Stroke and hairline weights

`02_VISUAL_LANGUAGE.md` §1 and `06_COMPONENT_LANGUAGE.md` establish hairline borders as the primary boundary device (replacing filled cards) without stating a weight, and `12_ACCESSIBILITY.md` §1 requires that weight to be verifiably tested against contrast minimums — which requires a concrete value to test in the first place.

- **1px hairline** — the default weight for every UI border: tag outlines, table dividers, panel boundaries, footer column rules. Always 1px regardless of display density (never a device-pixel-ratio-scaled "thicker on retina" hairline, which would break the deliberate consistency of the device).
- **1.5px** for diagram linework (`02_VISUAL_LANGUAGE.md` §9–11, `06_COMPONENT_LANGUAGE.md` §15) — slightly heavier than a UI hairline specifically so a diagram's real connective lines remain legible at the Drafting tier's larger scale (`05_LAYOUT_SYSTEM.md` §1) and hold up under `12_ACCESSIBILITY.md` §1's non-text contrast minimum at small diagram scales, where a 1px line would risk falling below it.
- **2px** for focus rings exclusively — deliberately the heaviest hairline weight on the site, because a focus indicator is the one hairline whose entire job is being unambiguous under `12_ACCESSIBILITY.md` §5's visibility requirement, not quiet.
- **Never a weight below 1px** (a sub-pixel hairline that renders inconsistently across browsers/zoom levels) and never a soft/blurred border simulating a hairline — every stroke in this system is crisp, per the drafting-sheet metaphor's own logic (a real technical drawing's linework is precise, not soft).

## 5. Elevation

`04_COLOR.md` §7 establishes that shadows stay functional and never carry the accent hue, without specifying a scale. Two steps, not a large system, because Working Blueprint's flat, panel-based composition (`06_COMPONENT_LANGUAGE.md` §2) needs elevation far less often than v2's card-based system did:

- **Resting (0):** no shadow — the default state for the vast majority of surfaces (panels, sections, the page itself), consistent with cards being retired as a default pattern.
- **Raised (1):** a low, tight shadow (small blur radius, low spread, low opacity — calibrated per theme per `04_COLOR.md` §7) for the small set of surfaces that need to visually lift off the base: an open dropdown, a modal, a tooltip, a Studio-register (`06_COMPONENT_LANGUAGE.md` §0) utility panel.
- **No third elevation step.** A system with more than two shadow states starts implying a z-axis hierarchy Working Blueprint's flat material logic doesn't have — real drafting sheets and real PCBs are flat; only a genuinely floating UI element (something that overlaps other content and must be read as being "above" it) earns the raised step at all.

## 6. Icon system

The single highest-severity gap `CRITIQUE_01_REVIEW_BOARD.md` §39 found in `02_VISUAL_LANGUAGE.md` — "no actual icon set specified... five different engineers... will invent five different 'diagonal line' icon styles." This document specifies the actual minimum icon set every implementation needs, plus the construction rule that keeps future additions consistent.

**Construction rule** (governs every icon, existing or future): drawn on a 24×24px grid, 1.5px stroke weight (matching diagram linework, §4), right angles and 45-degree diagonals only — literally no curves, no circles-as-primitives (a status dot is a small square or diamond, per `06_COMPONENT_LANGUAGE.md` §12's timeline-node precedent, never a circle) — because a curved icon next to right-angle trace-path linework elsewhere on the page would be the one visible inconsistency in an otherwise disciplined system.

**The minimum required set — every icon a real HubZero page needs today, specified so no engineer has to invent one:**

| Icon                                                 | Construction                                                                                                                                                                                                                                                                                 |
| ---------------------------------------------------- | -------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Menu (hamburger)                                     | Three horizontal traces, equal length, equal gap (the spacing scale's step 0)                                                                                                                                                                                                                |
| Close                                                | Two 45° diagonal traces crossing at their midpoint — literally an "X" drawn in the system's own diagonal-trace geometry, not a rounded glyph                                                                                                                                                 |
| Chevron (expand/collapse, accordion)                 | Two traces meeting at a right angle, rotated to point the required direction — never a curved arrowhead                                                                                                                                                                                      |
| External link                                        | A square outline with a 45°-diagonal trace exiting its top-right corner, echoing a real schematic "off-page connector" convention                                                                                                                                                            |
| Search                                               | A square (not circular) magnifier frame with a 45°-diagonal handle — the one deliberate departure from the literal "no circles" rule other systems use for a magnifier is resolved by squaring the lens itself, keeping the icon in-system                                                   |
| Filter                                               | Three horizontal traces of decreasing length with small square nodes on each — directly borrowed from the timeline-node convention (`06_COMPONENT_LANGUAGE.md` §12) so a filter control visually rhymes with a real diagram element rather than looking like a generic settings-slider glyph |
| Status indicator (active/live)                       | A small filled square node, matching a diagram's junction convention exactly — deliberately the same shape as a timeline milestone node (`06_COMPONENT_LANGUAGE.md` §12), reinforcing that a status dot and a diagram node are, in this vocabulary, the same kind of object                  |
| Copy-to-clipboard                                    | Two overlapping squares offset at a right angle — no rounded-corner "pages" glyph                                                                                                                                                                                                            |
| Success / error / warning (form and status feedback) | A square, diamond, and triangle respectively (echoing real schematic severity/shape conventions), each always paired with the semantic color and a text label per `12_ACCESSIBILITY.md` §2 — never color or shape alone                                                                      |

**Every icon in this table ships with its accessible name specified here, not left to an implementer:** Menu → "Open menu"; Close → "Close"; Chevron → "Expand"/"Collapse" (contextual); External link → "Opens in a new tab"; Search → "Search"; Filter → "Filter results"; Status indicator → a contextual live-status label (e.g. "Live demo available"), never a bare icon with no name; Copy-to-clipboard → "Copy to clipboard"; Success/error/warning → the semantic word itself ("Success", "Error", "Warning"), redundant with but never replaced by the paired text label per `12_ACCESSIBILITY.md` §2 and §5.

**A ninth icon is added only when a real page genuinely needs one no entry above covers** — the same closed-vocabulary discipline `ARCHITECTURE/20_CONTENT_BLOCKS.md` §1.1 already applies to CMS block types, applied here to icons: a proposed new icon should clear the same bar these nine did (a real, recurring UI need, not a nice-to-have), and should be added to this table, in this document, not invented ad hoc inside a component's implementation.

## 7. Image aspect ratios

`07_IMAGERY.md` specifies imagery categories and placement without stating crop ratios, which `02_VISUAL_LANGUAGE.md` §7's "one crop ratio per category, applied identically to every instance" rule requires in order to be enforceable at all.

| Category                                                                               | Ratio                                                                                                                                                                                                                                                                | Rationale                                                                                                                             |
| -------------------------------------------------------------------------------------- | -------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------- |
| Team/founder photography (`07_IMAGERY.md` §6, §14)                                     | 4:5 (portrait)                                                                                                                                                                                                                                                       | A tighter, more considered crop than a square — reads as a real portrait sitting, not a generic square avatar grid                    |
| Hardware/PCB macro photography (`07_IMAGERY.md` §4, §6)                                | 4:3                                                                                                                                                                                                                                                                  | Enough width to show real board context around a detail shot, without the extreme width a cinematic ratio would force onto a close-up |
| Product renders / screenshots (`07_IMAGERY.md` §2)                                     | 16:9 for a full-bleed hero instance, 4:3 for an inline supporting instance — the two-ratio split is deliberate, not an inconsistency, because a hero and an inline figure have genuinely different jobs (climax vs. supporting evidence, `05_LAYOUT_SYSTEM.md` §6–8) |
| Architecture/system diagrams (`07_IMAGERY.md` §3, `06_COMPONENT_LANGUAGE.md` §15)      | No fixed ratio — a diagram's shape is dictated by what it depicts, never cropped to an arbitrary frame the way a photograph is                                                                                                                                       |
| Gallery/contact-sheet grids (`06_COMPONENT_LANGUAGE.md` §16, `07_IMAGERY.md` §10, §13) | 1:1 (square) per cell, consistent across every cell in one gallery — the one place a square crop is correct, specifically because a contact-sheet convention depends on uniform cells                                                                                |
| Case-study cover image (`09_PAGE_ARCHETYPES.md` §5)                                    | 3:2                                                                                                                                                                                                                                                                  | Wide enough for a genuine "evidence" crop, narrower than 16:9 so it reads as a photograph/render rather than a cinematic banner       |
| Work/Labs/Blueprints index entry thumbnail (`09_PAGE_ARCHETYPES.md` §4)                | 4:3, consistent across all three index types, per `07_IMAGERY.md` §6's cross-index consistency requirement                                                                                                                                                           |                                                                                                                                       |

**A category with no real image yet ships with no image, per `07_IMAGERY.md` §1** — this table specifies the ratio a future real image must be cropped to, never a placeholder occupying that ratio in the meantime.

## 8. Governance — how this table changes

Per `CRITIQUE_01_REVIEW_BOARD.md` Part 8, finding 3 (design token governance, named as missing and not addressed by any of the original ten documents): a value in this document changes only when a real, named page or component genuinely cannot be built within it — never speculatively "for flexibility." The process is deliberately lightweight, matching this being a five-person studio rather than a large organization with a formal design-ops function: propose the change against the specific real need in the same pull request that needs it, name which section of this document it amends, and update the table here in the same change — never a silent drift where an implementation quietly uses a value this document doesn't list. A token that needs to be added, not just tuned within an existing range, should be rare enough that it's worth noticing when it happens.
