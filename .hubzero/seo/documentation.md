# Documentation SEO

Product documentation and user guides build visibility by answering the exact question a user has at the exact moment they have it.

Documentation SEO succeeds when a user finds the answer without needing to contact support, and fails silently every time they can't and never tell anyone.

---

# Search Intent

Visitors typically search to:

- Learn how to accomplish a specific task
- Resolve an error or unexpected behavior
- Understand a concept before using a feature
- Confirm whether a feature exists or behaves a certain way

---

# Primary Keyword Categories

- Task-oriented phrases ("how to …", "how do I …")
- Feature and concept names, in the exact terms the product uses
- Error messages and error codes, verbatim
- Comparative and troubleshooting phrases

---

# Essential Pages

Required

- Getting started / quickstart
- Core concepts
- Task-oriented guides for primary workflows
- Troubleshooting / FAQ
- Search results page that actually works

Recommended

- Changelog or release notes
- Glossary of product-specific terms

---

# Information Architecture

Structure documentation around user tasks and concepts, not around internal product or team structure. A user should never need to know which team built a feature to find out how to use it.

Maintain one canonical page per task or concept. Duplicate or near-duplicate pages fragment both search ranking and user trust in which page is current.

---

# Internal Linking Strategy

Link concepts to the tasks that use them, and tasks to the concepts they assume. Link troubleshooting content back to the relevant task page rather than leaving it isolated.

---

# Structured Data

Recommended

- BreadcrumbList
- FAQPage, where content is genuinely structured as Q&A
- TechArticle or Article, where applicable

---

# Content Strategy

Write for the reader's task, not for the feature's implementation. Keep every page's scope narrow enough that it can be kept accurate as the product changes — a page trying to cover too much becomes the first page that goes stale.

Version documentation explicitly when the product has multiple supported versions. Never let a user land on instructions for a version they aren't using without a clear signal.

---

# Technical Considerations

Fast search, accurate site search indexing, clean URLs that survive page reorganization, canonical URLs when content is mirrored across versions, sitemap coverage for every published page.

---

# Common Mistakes

- Organizing docs by internal team or system architecture instead of user task
- Multiple pages answering the same question with subtly different, drifting answers
- No visible way to tell which product version a page describes
- Search that only matches exact titles, not task phrasing
- Letting docs silently go stale after a feature changes

---

# Success Metrics

Search-to-resolution rate, reduction in related support tickets, time on page balanced against task completion (not simply maximized), internal search success rate.
